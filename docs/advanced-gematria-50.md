## פונקציות מנוע-העל (מעבר לשיטות)
1. **כל-השיטות-במבט-אחד** — טבלת 50 הערכים בו-זמנית למילה.
2. **חיפוש הפוך** — ערך → כל הביטויים/השמות במאגר שמגיעים אליו (בכל שיטה).
3. **מטריצת-הצלבות** — שתי מילים × 50 שיטות → היכן הן נפגשות.
4. **ניתוח אות-אות** — פירוק מלא לכל שיטה.
5. **שמירת-מחקר + ייצוא** (CSV/תמונה) + Quick/ToolActions.
6. **קרדיטים** — חישוב-מתקדם/דוח = X קרדיטים (`platform_tiers_law`).

## ארכיטקטורה (נעול — מודל «3 שכבות», אישור צוריאל)
> צוואר-הבקבוק העתידי = מנועי-החיפוש (ELS · הצלבות · AI · גרפים), **לא** הגימטריה.
1. **מילים — פעם אחת.** `gematria_words(id, text, …)`. הוספת שיטות **לא נוגעת** בטבלה הזו.
2. **Registry של שיטות — הגדרה בלבד, לא דאטה.** מימוש = פונקציה אחת ב-`src/lib/gematria.js` לפי `key`. ההגדרה הנעולה (האמת) = `nodes type=rule`. טבלת-מטא אופציונלית `method_registry(key, name, family, priority, tier, credits)` — מצביעה על הפונקציה, **לא** מאחסנת ערכים.
3. **חישוב בזמן-אמת** (`calcGem`, client-side) ברוב המקרים — זול מ-round-trip ל-DB. **בלי cache לגימטריה.**
4. **Cache רק ליקרים:** ELS · חיפוש-הפוך בכל המאגר · «כל המילים בערך X» · גרפי-קשרים · AI · הצלבות-המוניות. שם משקיעים אופטימיזציה.
- תוצאה: גם ב-80–100 שיטות הארכיטקטורה נשארת נקייה — מוסיפים הגדרה+פונקציה, אפס שכפול-דאטה.

### תהליך-העבודה (נעילה לפני קוד)
1. צוריאל עובר משפחה-משפחה, מאשר/מתקן/נועל כל הגדרה.
2. כל שיטה מאושרת → `nodes type=rule` (rule_id ייחודי) + מימוש ב-`src/lib/gematria.js`.
3. שכבת-הגישה (tier) נבנית מעל `profiles`.
4. UI מנוע-העל = הרחבת `GematriaCalculator` במצב «מתקדם» (לא מסך חדש).
